# Prompt library — Eval Forge

Reusable prompts threaded through the Eval Forge workflow. Copy, adapt, ratify. Each prompt assumes you've already read `CLAUDE.md` for the project at hand. Because Forge's job is to grade other features, its own prompts are versioned in `prompts/` and gated by the meta-eval scorer.

## 01 — Plan a generator improvement

Use at the start of every change to the generation pipeline. The agent outputs a numbered task list; you edit before any code is touched.

> Read `CLAUDE.md` and the relevant section of `tech-spec.mdx`. Then, for change `<NAME>`, draft a numbered task list with concrete file paths.
>
> Each task must be:
>
> - ≤ 1 hour of agent work,
> - independent enough to land as its own commit,
> - written as an imperative ("Add X", "Refactor Y").
>
> End with a verification step listing the commands (including `pnpm test:meta`) and manual checks that prove the change is done.

## 02 — Generate eval cases from a spec section

The core prompt. Versioned in `prompts/generate-cases.ts`; every change ships paired with a meta-eval-score delta.

> You are an eval-case author. Given the feature spec below, produce exactly 10 evaluation cases as JSON.
>
> Each case must include:
>
> - `id` — short kebab-case identifier,
> - `input` — the exact input the feature would receive,
> - `expected` — the desired output, OR a rubric specification if exact match isn't appropriate,
> - `kind` — one of `exact-match`, `rubric-score`, `human-review`,
> - `notes` — one sentence explaining what this case is testing.
>
> Distribution: 5 happy paths, 3 boundary cases, 2 hostile inputs. No emoji. No marketing prose.

## 03 — Rubric design for a fuzzy output

Use when the feature's output isn't deterministic and rubric-score is the right kind.

> Design a 3–5 axis rubric for evaluating outputs of feature `<NAME>`. Each axis must be:
>
> - independent (low correlation with the others),
> - scorable on a 0–1 continuous scale with a Claude judge,
> - tied to a concrete user-visible behavior (no "quality" abstractions).
>
> Output as a TypeScript const matching the `Rubric` type in `packages/harness/rubric.ts`.

## 04 — Audit a generated suite

Use after a generator run, before committing the output.

> Audit the generated suite at `./forge-output/`. Answer:
>
> 1. Are happy-path cases meaningfully different from each other, or is the model repeating itself?
> 2. Are boundary cases actually at the boundary, or are they just middle cases relabeled?
> 3. Is any case ambiguous enough that a reasonable model could pass or fail on tone alone?
>
> Suggest fixes; do not apply them.

## 05 — Write release notes from a meta-eval delta

Use after merging a prompt change to main, before promoting the generator to production.

> Given this prompt diff plus the meta-eval score table (before vs. after across the reference pairs), draft user-facing release notes:
>
> - One sentence summary at the top.
> - Two sections: **Improved** (axes where the generator got sharper), **Watch** (anything that regressed even slightly).
> - Each entry is a single line, user-facing language, no jargon.
> - Skip anything purely internal to the harness.

## 06 — Refactor on the third repetition

Use when you spot the same shape three times.

> Three places in this codebase do `<DESCRIBE THE SHAPE>`:
>
> - `<file:line>`
> - `<file:line>`
> - `<file:line>`
>
> Propose a single shared abstraction. Show the proposed signature and call sites. Do not apply the change — I want to see the shape first.
