# PRD — Eval Forge

## 1. Problem

Shipping a Claude-powered feature without an eval suite is shipping on vibes. Most teams know this, and most teams ship without one anyway — the activation energy is too high to hand-author eval cases at the moment of feature work, and the harness scaffolding is a half-day sidequest no one wants. So the suite never gets written, the prompt drifts, and the regression ships to users before anyone notices.

The eval gate should already exist.

## 2. Target user

- **Primary**: engineers shipping Claude features inside a TypeScript codebase, who already use Vitest.
- **Secondary**: PMs and tech leads who want a numeric grade per feature to gate merges with.

JTBD: _"When I add a Claude feature, I want a passing-or-failing eval gate behind it before the PR opens — without writing the harness myself."_

## 3. Hypothesis

> If we hand a user a runnable Vitest eval suite + golden set + GitHub Action from a spec section, they will gate at least 70% of Claude-feature PRs behind it within 30 days of installation.

**Success metric**: % of Claude-feature PRs in the target repo with a passing forge-generated eval check ≥ 70% within 30 days. (Baseline: most teams gate 0% — the suite doesn't exist.)

## 4. MVP scope

**In**:

- CLI: `forge generate ./spec.md --feature user-bio-rewrite` emits the test suite + goldset + Action.
- Web dashboard: paste a spec section, get the same output as a downloadable zip.
- 3 eval kinds at MVP: exact-match, rubric-score (Claude-as-judge), and human-review queue.
- A meta-eval scorer so the generator's own output quality is measurable.

**Out** (post-MVP):

- Multi-feature batching ("eval my whole codebase").
- Non-TypeScript harness targets (Python, Go).
- Self-hosted dashboard.
- Live dashboards for non-CI runs.

## 5. User flows

1. `forge generate ./spec.md --feature <name>` → suite, goldset, and `forge.yml` written to `./forge-output/`.
2. Review the goldset cases, hand-edit the 1–2 that need it, commit alongside the feature.
3. Open the PR → the generated Action runs `vitest`, posts a pass/fail check with per-axis scores.
4. Dashboard alternative: paste the same spec at `forge.blokz.dev`, download the zip, drop it in.

## 6. Constraints

- **Perf**: a 10-case suite generated in ≤ 20s on the CLI; dashboard streams the first case in ≤ 3s.
- **Determinism**: regenerating from the same spec yields a stable file layout (cases may differ; structure must not).
- **Cost**: < \$0.10 of API spend per generated suite at MVP case counts.
- **Fidelity**: the generator must hold its meta-eval score ≥ 0.85 before any prompt change ships.
- **Trust**: Claude-as-judge is never the _only_ gate — a human-review queue ships alongside it.

## 7. Distribution per platform

| Platform | Channel                                          | Beta cohort        |
| -------- | ------------------------------------------------ | ------------------ |
| Web      | forge.blokz.dev, edge-streamed generation        | invite link        |
| Windows  | `cargo install forge-cli` + winget manifest      | CLI early adopters |
| Android  | Play Console internal track — read-only run view | 25 engineers       |
| iOS      | TestFlight cohort — read-only run view           | 25 engineers       |

## 8. Risks

- **Generated suites are noisy** → mitigate with the meta-eval scorer; tune prompts on a curated reference set before any release.
- **Claude-as-judge bias** → emit a human-review queue alongside, never as the only gate; surface the judge's reasoning per case.
- **Spec sections vary wildly in quality** → the CLI refuses to generate from a spec under a length/structure threshold and points to a spec template.
- **Harness lock-in fear** → emitted output is plain Vitest + JSON; nothing imports a Forge runtime, so a team can delete Forge and keep its suites.

## 9. Open questions

- Generate the GitHub Action by default, or behind a `--ci` flag? Lean: behind a flag — not every repo wants it on the first run.
- Ship a VS Code extension at MVP, or CLI + dashboard only? Lean: CLI + dashboard; the editor surface is post-MVP.
- Cap generated cases at 10, or scale with spec length? Lean: cap at 10 for MVP; predictable cost beats coverage early.
