# Prompt library

Reusable prompts threaded through the Blokz vibecoding workflow. Copy, adapt, ratify. Each prompt assumes you've already read `CLAUDE.md` for the project at hand.

## 01 — Plan a feature

Use at the start of every new feature. The agent outputs a numbered task list; you edit before any code is touched.

> Read `CLAUDE.md` and `content/workflow/phases.ts`. Then, for feature `<NAME>`, draft a numbered task list with concrete file paths.
>
> Each task must be:
>
> - ≤ 1 hour of agent work,
> - independent enough to land as its own commit,
> - written as an imperative ("Add X", "Refactor Y").
>
> End with a verification step listing the commands and manual checks that prove the feature is done.

## 02 — Audit a file

Use before refactoring or when a file feels heavy.

> Audit `<PATH>` for the following:
>
> 1. Unused exports / dead code.
> 2. Dead branches (unreachable conditionals).
> 3. Violations of the conventions in `CLAUDE.md` §5.
> 4. Code that would surprise a fresh reader.
> 5. Hard-to-test seams (functions doing too much).
>
> Suggest fixes; **do not apply them**. Output as a bulleted list with file:line citations.

## 03 — Write a PR description from a diff

Use after the implementation is done and tests are green.

> Read this diff and write a PR description in our voice. Use the structure below — no emoji, no marketing-speak.
>
> 1. **Title** — imperative, ≤ 72 chars.
> 2. **Summary** — one paragraph: what changed, why now.
> 3. **What changed / what didn't** — two short bulleted lists.
> 4. **Screenshot** — `[placeholder]` for the human to fill.
> 5. **Verification** — checklist of commands and manual checks that pass.

## 04 — Generate test cases from a spec

Use when adding a new feature spec'd in `tech-spec.mdx`.

> Given this spec section, generate a Vitest test plan as a list of `describe` and `it` headings only.
>
> Include:
>
> - happy paths,
> - boundary cases,
> - error states from the spec's error matrix,
> - one explicit **non-goal** to make the scope clear,
> - one perf assertion (latency budget).
>
> Do not write the test bodies — only the structure.

## 05 — Refactor on the third repetition

Use when you spot the same shape three times.

> Three places in this codebase do `<DESCRIBE THE SHAPE>`:
>
> - `<file:line>`
> - `<file:line>`
> - `<file:line>`
>
> Propose a single shared abstraction. Show the proposed signature and call sites. Do not apply the change — I want to see the shape first.

## 06 — Write release notes from the diff

Use after merging to main, before promoting to production.

> Given this commit list and the PR descriptions linked, draft user-facing release notes:
>
> - One sentence summary at the top.
> - Three sections: **New**, **Improved**, **Fixed**.
> - Each entry is a single line, user-facing language, no jargon.
> - Skip anything that is purely internal.
