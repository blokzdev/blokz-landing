# PRD — Blokz Receipt

## 1. Problem

Block explorers are built for engineers. Accountants reconciling crypto invoices and founders defending grant spends have to translate a 20-field transaction page into a paper trail by hand — copy the hash, screenshot the explorer, label the counterparty, add a memo, export to PDF, attach to the email.

The receipt should already exist.

## 2. Target user

- **Primary**: small-team accountants and founders managing crypto payables under \$500K/yr.
- **Secondary**: power users sharing donation receipts, contractors invoicing in stables.

JTBD: _"When I pay someone in crypto, I want a one-page receipt I can hand my accountant — without copy-pasting eight fields."_

## 3. Hypothesis

> If we hand a user a one-page printable receipt for a tx, they will share it 3× more than they share a raw block-explorer URL.

**Success metric**: shares per session ≥ 0.4 within 30 days of launch. (Baseline: explorer share rate is 0.13.)

## 4. MVP scope

**In**:

- Single input: tx hash _or_ block-explorer URL (auto-detect chain).
- Multi-chain: Bitcoin, Ethereum, Polygon, BNB Chain.
- Editable counterparty label + memo (stored locally, never sent).
- PDF + PNG export.
- Native share sheet on mobile.

**Out** (post-MVP):

- Multi-tx batching for invoices.
- Wallet connect for on-chain "signed" receipts.
- Tron / Solana support.
- Custom logo upload.

## 5. User flows

1. Land on `/` → paste hash → instant receipt preview (≤ 1s).
2. Optional: edit counterparty + memo → live update.
3. Tap **Share** → native share sheet (`web.share()` on mobile, fallback to "Copy link" + "Download PDF" on desktop).
4. Recipient opens link → same receipt, same fields, no re-render delay (edge-cached).

## 6. Constraints

- **Perf**: LCP ≤ 1.5s on 3G mobile.
- **A11y**: WCAG 2.1 AA, full keyboard nav, reduced-motion safe.
- **Privacy**: zero analytics on hash content; user-typed labels never leave the device.
- **Cost**: < \$50/mo infra for first 10K WAU.

## 8. Risks

- **Explorer API rate limits** → mitigate with a Supabase tx cache (5 min TTL, indefinite for confirmed txs).
- **Trademark concerns** around "Receipt" → confirmed generic; no claim.
- **Phishing reuse**: bad actor sends a forged "receipt link." Mitigation: every receipt links to source explorer; never claims to confirm payment in the marketing copy.

## 9. Open questions

- Should we render the receipt server-side as PDF, or client-side via the browser print stylesheet? (Lean: client-side, simpler infra.)
- Do we want a `receipt.blokz.dev` subdomain, or live under the main marketing domain? (Lean: subdomain, easier scope.)
- Multi-language at MVP, or post-launch?
