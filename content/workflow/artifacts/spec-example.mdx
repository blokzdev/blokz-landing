# Tech spec — Blokz Receipt

## 1. Architecture

```txt
User
 ↓ paste hash
Next.js App Router (RSC)
 ↓
Edge function: /api/receipt
 ↓ cache hit? → respond
 ↓ cache miss
Blockchair API (or per-chain adapter)
 ↓
Supabase cache (5 min TTL, ∞ for confirmed)
 ↓
React render → ReceiptCard
 ↓
Print stylesheet → PDF
 ↓
web.share() / Copy link / Download PNG
```

## 2. Data shapes

```ts
type Chain = "bitcoin" | "ethereum" | "polygon" | "bsc";

interface Receipt {
  hash: string;
  chain: Chain;
  from: string;
  to: string;
  value: bigint; // base units (wei, sat)
  fee: bigint;
  timestamp: number; // Unix seconds
  blockHeight: number;
  status: "success" | "fail" | "pending";
  /** User-typed, stored in localStorage only, never sent. */
  labels?: { counterparty?: string; memo?: string };
}
```

## 3. API surface

### `POST /api/receipt`

Edge runtime.

**Request**:

```json
{ "hash": "0xabc...", "chain": "ethereum" }
```

**Response** (200):

```json
{
  "hash": "0xabc...",
  "chain": "ethereum",
  "from": "0x...",
  "to": "0x...",
  "value": "1500000000000000000",
  "fee": "210000000000000",
  "timestamp": 1715000000,
  "blockHeight": 19450000,
  "status": "success"
}
```

**Errors**: 400 (invalid hash), 404 (not found), 429 (rate-limited), 502 (upstream).

### `GET /api/og/[hash]`

ImageResponse — generates a 1200×630 social-share card with the receipt fields rendered.

## 4. State management

- **Server**: receipt data fetched in RSC.
- **Client**: user labels (counterparty, memo) — `useState` + `localStorage`. No global store.
- **URL**: hash + chain as path params. No query strings.

## 5. Error states

| Case                     | UX                                                    |
| ------------------------ | ----------------------------------------------------- |
| Invalid hash format      | Inline form error, no API call                        |
| Chain not supported      | Suggest closest match ("Did you mean Ethereum?")      |
| API timeout (3s)         | Retry once, then degraded mode (Supabase cache only)  |
| Pending tx               | Receipt shows yellow "pending" badge, polls every 15s |
| Failed tx                | Receipt shows red "failed" badge with reason          |
| Upstream API unavailable | Friendly error + link to source explorer              |

## 6. Security & privacy

- **No persistent storage** of viewed hashes (cache is keyed by hash but anonymized — no IP, no user-id).
- **Rate limit** `/api/receipt`: 30 req/min/IP (Upstash Rate Limiting on Edge).
- **User labels** never leave the device. `localStorage` only.
- **No third-party analytics**. Vercel Analytics privacy-respecting only.
- **CSP**: `default-src 'self'`; explicit image hosts for explorer favicons.

## 7. Test strategy

- **Vitest unit**: chain adapters (one suite per chain), formatters, hash validators.
- **Playwright e2e**: paste → render → share flow on Chrome, Firefox, mobile Safari.
- **Visual regression**: receipt card across 4 chains × success/fail/pending = 12 snapshots.
- **Load**: k6 smoke against `/api/receipt` at 100 req/s steady state.

## 8. Rollout

| Stage         | Audience        | Gate                         |
| ------------- | --------------- | ---------------------------- |
| Internal beta | 20 testers      | Manual QA pass               |
| 5% canary     | Random sampling | 1 week clean Sentry          |
| 100%          | All             | Kill-switch flag operational |

## 9. Telemetry events

Minimal, all anonymous:

- `receipt_rendered` — chain, latency bucket
- `receipt_shared` — share method (native / copy / pdf / png)
- `receipt_error` — error code, chain
